## How I Work

I learn by building.

Most of my projects start from a real security problem, a manual workflow, or a gap I have seen in existing tooling. I then break the problem down, build a proof of concept, test it in a lab, and gradually turn it into something more structured.

My approach is:

1. Understand the problem properly
2. Build a working proof of concept
3. Test it against realistic scenarios
4. Improve security, usability, and reliability
5. Document the tool clearly
6. Keep iterating until it becomes genuinely useful

I prefer practical security engineering over theory alone. If a tool does not help someone detect, understand, prevent, or respond to a real issue, it needs more work.
